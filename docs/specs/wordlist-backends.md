# Word List Backends — Future Enhancement (DRAFT)

> **Status**: Draft — speculative. Not scheduled for implementation.

## Context

The gematria CLI currently supports file-based word lists (plain text and TSV).
The core package defines a `WordSource` interface for reverse lookups, with a
single in-memory implementation backed by `ParseWordList(io.Reader)`.

This document captures the intent behind that interface and outlines potential
future backends.

## Why an Interface

Reverse lookup (`FindByValue`) accepts a `WordSource` interface rather than a
concrete `[]Word` slice. This allows alternative backends to be introduced
without changing the public API signature. The in-memory implementation
remains the default and only implementation for now.

## Potential Backends

### Embedded Database (e.g., SQLite)

- Pre-indexed word lists for faster lookups on large corpora.
- Avoids loading the entire word list into memory.
- Could support multiple word lists in a single database file.
- The CLI would accept a `.db` file via `--wordlist` the same way it accepts
  `.tsv` files today. Backend selection based on file extension or content
  detection.

### Pre-Computed Index File

- A companion index file (e.g., `words.tsv.idx`) mapping gematria values to
  file offsets.
- Faster than linear scan for repeated lookups against large files.
- Generated by a `gematria index` subcommand or on first use.

### Remote / API Source

- A `WordSource` backed by an HTTP API or gRPC service.
- Useful for shared curated word lists hosted centrally.
- Introduces network dependency, authentication, caching concerns.

## Suggestions

- Keep the `WordSource` interface minimal. The current shape should be:
  a method that accepts a target value, a gematria system, and a limit,
  and returns matching words. Resist adding query methods until a real
  backend demands them.
- The in-memory implementation should remain the reference. New backends
  must produce identical results for the same input data.
- Backend selection belongs in `internal/cli/`, not the root package. The
  domain accepts a `WordSource`; the CLI decides which implementation to
  construct based on flags, file extensions, or configuration.

## Open Questions

1. **Should `WordSource` support streaming results?** The current design
   returns `[]Word`. For very large result sets, a channel or iterator
   pattern might be more memory-efficient. But the `--limit` flag caps
   results at a modest number, making streaming unnecessary for now.

2. **Should the interface include a `Close()` method?** Database and remote
   backends need cleanup. Adding `io.Closer` to the interface would force
   the in-memory implementation to have a no-op `Close()`. Alternatively,
   backends that need cleanup can implement `io.Closer` separately, and the
   CLI can check for it with a type assertion.

3. **Should indexing be automatic or explicit?** If a pre-computed index
   backend is added, should the CLI build the index on first use
   (transparent but slow on first run) or require an explicit
   `gematria index --wordlist ${WORDLIST}` step?

4. **Should the `WordSource` interface support listing all words?** Some
   consumers might want to iterate the full list (e.g., for export or
   analysis). This is outside the current requirements but would be natural
   for a database backend.

5. **Multiple word lists**: Should `FindByValue` support searching across
   multiple word sources simultaneously? Or is that a CLI-level concern
   (call `FindByValue` once per source, merge results)?
