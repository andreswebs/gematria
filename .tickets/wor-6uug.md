# Pluggable Word List Backends

Implement alternative WordSource backends beyond the current in-memory implementation, as described in docs/specs/wordlist-backends.md. The WordSource interface (defined in the reverse-lookup epic) accepts any conforming backend, so all new backends slot in without changing the root package API. Backend selection lives entirely in internal/cli/; the domain layer is unaffected.

## Tasks

1. sqlite-backend — SQLiteWordSource implementation in src/backend_sqlite.go; also drives the gematria index subcommand for SQLite output
2. index-file-backend — IndexWordSource for pre-computed sorted TSV index in src/backend_index.go
3. index-subcommand — gematria index CLI subcommand (dispatched before flag parsing) to generate .db or .idx files from a plain/TSV word list
4. remote-backend — RemoteWordSource backed by an HTTP API in src/backend_remote.go
5. backend-selection — Auto-detection of backend type in internal/cli/run.go, io.Closer cleanup via type assertion, and optional --wordlist-format override flag
6. backend-tests — Unit and CLI integration tests for all three new backends; reference in-memory implementation as the correctness oracle

## Full Implementation Plan

### Design Constraints (from wordlist-backends.md)

- WordSource interface stays minimal: FindByValue(value int, system System, limit int) ([]Word, bool, error). No new methods added until a real backend demands them.
- Backends that need cleanup implement io.Closer separately; CLI checks with a type assertion after constructing the source and calls defer closer.Close() if the assertion succeeds.
- io.Closer is NOT embedded in the WordSource interface itself.
- Backend selection belongs in internal/cli/, not the root package.
- The in-memory implementation (ParseWordList) remains the reference; new backends must produce identical results for the same input data.
- Root package never imports "os" or net/http — backends are domain types, not CLI types.

### Root Package Files

#### src/backend_sqlite.go

Dependency: modernc.org/sqlite (CGo-free pure-Go SQLite driver).

Schema (created by gematria index, not by the backend itself):
  CREATE TABLE words (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    hebrew TEXT NOT NULL,
    transliteration TEXT NOT NULL DEFAULT '',
    meaning TEXT NOT NULL DEFAULT ''
  );
  CREATE TABLE word_values (
    word_id INTEGER NOT NULL REFERENCES words(id),
    system TEXT NOT NULL,
    value INTEGER NOT NULL
  );
  CREATE INDEX idx_word_values ON word_values(system, value);

Exported constructor (returns interface to hide implementation):
  func OpenSQLiteWordSource(path string) (WordSource, error)

Unexported struct:
  type sqliteWordSource struct { db *sql.DB }
  var _ WordSource = (*sqliteWordSource)(nil)
  var _ io.Closer  = (*sqliteWordSource)(nil)

FindByValue: SQL query joining words + word_values WHERE system=? AND value=?, LIMIT limit+1. Detect hasMore from len(rows) > limit.

Close: db.Close().

#### src/backend_index.go

Format: sorted TSV file, each line: <system>\t<value>\t<hebrew>\t<transliteration>\t<meaning>, grouped and sorted by (system, value). Generated by gematria index --format index.

On OpenIndexWordSource: single sequential scan builds map[System]map[int]indexRange{offset int64, count int} — only offsets and counts, not word data, keeping memory use tiny.

FindByValue: look up indexRange in the map (O(1)), seek to offset, read count+1 lines, parse Words, return results[:min(count,limit)], count > limit, nil.

Unexported struct:
  type indexWordSource struct {
    f     io.ReadSeeker // file kept open for deferred seeks
    index map[System]map[int]indexRange
  }
  var _ WordSource = (*indexWordSource)(nil)
  var _ io.Closer  = (*indexWordSource)(nil)

Exported constructor:
  func OpenIndexWordSource(path string) (WordSource, error)

Close: underlying file close.

#### src/backend_remote.go

API contract: GET {baseURL}/words?value=N&system=S&limit=L
Response: {"words":[{"hebrew":"...","transliteration":"...","meaning":"..."}],"hasMore":bool}

Unexported struct:
  type remoteWordSource struct {
    baseURL    string
    httpClient *http.Client
    authToken  string
  }
  var _ WordSource = (*remoteWordSource)(nil)

Exported constructor with functional options:
  func NewRemoteWordSource(baseURL string, opts ...RemoteOption) (WordSource, error)
  type RemoteOption func(*remoteWordSource)
  func WithAuthToken(token string) RemoteOption
  func WithHTTPClient(client *http.Client) RemoteOption

FindByValue: build URL, set Authorization header if token present, execute GET, decode JSON, return (words, hasMore, err). Non-200 → wrapped error with status code. Network failure → wrapped error.

No io.Closer needed (HTTP client has no cleanup). remoteWordSource does not implement io.Closer.

### CLI Layer

#### src/internal/cli/run.go — openWordSource helper

  func openWordSource(path string, getenv func(string) string) (gematria.WordSource, error)

Detection order:
  1. http:// or https:// prefix → NewRemoteWordSource; reads GEMATRIA_WORDLIST_TOKEN for optional auth
  2. .db extension → OpenSQLiteWordSource
  3. .idx extension → OpenIndexWordSource
  4. companion file path+".idx" exists → OpenIndexWordSource(path+".idx")
  5. default → open file, ParseWordList, close file immediately (in-memory)

After construction in Run():
  if closer, ok := src.(io.Closer); ok { defer closer.Close() }

New optional flag --wordlist-format (sqlite|index|remote|memory) allows explicit override of auto-detection. New env var GEMATRIA_WORDLIST_TOKEN (lazy-validated only when remote backend is selected).

#### src/internal/cli/run.go — index subcommand

Dispatch at the top of Run() before any flag parsing:
  if len(args) > 0 && args[0] == "index" { return runIndex(args[1:], ...) }

runIndex flags:
  --wordlist ${WORDLIST_PATH}  (required)
  --output ${OUTPUT_PATH}      (optional; defaults to wordlist+".db" for sqlite, wordlist+".idx" for index)
  --format sqlite|index        (optional; default: sqlite)

Exit codes: 0 success, 2 missing --wordlist, 3 I/O error, 1 parse/computation error.

Process: open --wordlist → ParseWordList → iterate all words → Compute for all 4 systems → write to output.

For SQLite: open DB, CREATE TABLE/INDEX if not exists, INSERT in a transaction.
For index: write sorted TSV lines (sort by system then value in memory before writing).

### Testing

Unit tests live in src/ alongside the implementation files. CLI integration tests live in src/internal/cli/.

Each new backend is tested against the in-memory ParseWordList implementation using the same input data to verify result equivalence (correctness oracle pattern).

Remote backend tests use httptest.NewServer to avoid real network calls.

SQLite and index backend tests use os.TempDir() for ephemeral files, cleaned up with t.Cleanup.
