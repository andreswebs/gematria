# Tests for alternative WordSource backends

Write unit tests for SQLiteWordSource, IndexWordSource, and RemoteWordSource, plus CLI integration tests for backend auto-selection and the gematria index subcommand. The in-memory ParseWordList implementation is the correctness oracle: for any input data, every backend must return identical results.

## Spec References
- docs/specs/code-architecture.md §"Testing strategy" (root package: pure unit tests; CLI: integration tests with injected I/O)
- docs/specs/wordlist-backends.md §"Suggestions" (new backends must produce identical results for the same input data)

## Implementation Details

### src/backend_sqlite_test.go

Setup helper (shared across subtests):
  func buildTestDB(t *testing.T, words []gematria.Word) string
  Creates a temp SQLite DB in t.TempDir(), inserts all words with computed values for all 4 systems, returns the DB file path.

Test cases:
- TestSQLiteWordSource_FindByValue_MatchesInMemory: generate 50 random valid Hebrew words, build both an in-memory source (ParseWordList) and a SQLite source from the same data, call FindByValue for several (value, system) pairs, assert identical []Word results and hasMore values.
- TestSQLiteWordSource_FindByValue_HasMore: insert limit+5 words all sharing the same computed value, verify hasMore=true and len(results)==limit.
- TestSQLiteWordSource_FindByValue_NoMatch: query a value that no word has; expect (nil, false, nil).
- TestSQLiteWordSource_FindByValue_EmptyDB: open a valid but empty SQLite DB; expect (nil, false, nil) for any query.
- TestSQLiteWordSource_Close: call Close() then FindByValue; expect a non-nil error.

### src/backend_index_test.go

Setup helper:
  func buildTestIndex(t *testing.T, words []gematria.Word) io.ReadSeeker
  Generates an index file in the sorted TSV format (same as gematria index --format index), returns it as an in-memory bytes.Reader so no temp file is needed.

Test cases:
- TestIndexWordSource_FindByValue_MatchesInMemory: same oracle pattern as SQLite test.
- TestIndexWordSource_FindByValue_HasMore: limit+5 words with same value; verify hasMore.
- TestIndexWordSource_FindByValue_NoMatch: expect (nil, false, nil).
- TestIndexWordSource_FindByValue_EmptyIndex: only the magic comment line; expect (nil, false, nil).
- TestIndexWordSource_FindByValue_BoundaryValues: word with system's minimum possible value and maximum possible value; verify both found.

### src/backend_remote_test.go

Use httptest.NewServer to avoid real network calls.

Test cases:
- TestRemoteWordSource_FindByValue_QueryParams: capture the incoming request in the test server handler; assert query params value, system, and limit are correctly set.
- TestRemoteWordSource_FindByValue_AuthHeader: configure WithAuthToken("tok"); assert Authorization: Bearer tok header present.
- TestRemoteWordSource_FindByValue_ParseResponse: server returns a known JSON body; assert returned []Word matches expected.
- TestRemoteWordSource_FindByValue_HasMore: server returns hasMore:true; assert propagated.
- TestRemoteWordSource_FindByValue_Non200: server returns 404; assert non-nil error containing "404".
- TestRemoteWordSource_FindByValue_NetworkError: use a closed httptest server (server.Close() before calling FindByValue); assert wrapped network error.
- TestNewRemoteWordSource_InvalidBaseURL: empty string or "ftp://..." → constructor returns error.

### src/internal/cli/backend_integration_test.go

These tests call cli.Run() with injected I/O (same pattern as other CLI integration tests).

Shared fixture: a small TSV word list written to a temp dir.

Test cases:
- TestBackendSelection_DefaultInMemory: --wordlist points to plain .tsv file; verify --find returns correct results.
- TestBackendSelection_SQLite: --wordlist points to .db file (pre-generated by invoking runIndex internally or via a test helper); verify --find returns correct results.
- TestBackendSelection_IndexFile: --wordlist points to .idx file; verify --find returns correct results.
- TestBackendSelection_CompanionIndex: --wordlist points to plain .tsv but a matching .tsv.idx exists in the same dir; verify IndexWordSource is used (can be verified via an httptest hook or by checking the .idx file is opened via a custom io.ReadSeeker wrapper — or simply verify results match).
- TestBackendSelection_Remote: --wordlist is http://127.0.0.1:{port}/; httptest server answers; verify correct results.
- TestBackendSelection_ExplicitFormat: --wordlist-format=memory with a .db path → in-memory backend used; verify no SQL driver initialization.
- TestBackendSelection_InvalidFormat: --wordlist-format=bogus → exit code 2, error on stderr listing valid values.
- TestIndexSubcommand_SQLite: run gematria index --wordlist ${WORDLIST_PATH} → .db file created; open it with OpenSQLiteWordSource, verify FindByValue matches in-memory oracle.
- TestIndexSubcommand_IndexFile: run gematria index --wordlist ${WORDLIST_PATH} --format index → .idx file created; open with NewIndexWordSource, verify FindByValue matches oracle.
- TestIndexSubcommand_MissingWordlist: run gematria index with no --wordlist → exit code 2, error on stderr.
- TestIndexSubcommand_BadOutputPath: --output /nonexistent/dir/out.db → exit code 3.
